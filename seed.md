# Path.js `extname` Resolution Bug

## Проблема
При використанні абсолютних шляхів, які містять крапку в назві директорій (наприклад, `/Users/user/src/nan.web/apps/...`), метод `DBFS.loadDocument(absPath)` повертав `undefined`. 
Це ставалося через помилкову реалізацію функції `extname` у `packages/db/src/DB/path.js`. Стара реалізація робила простий `uri.split('.')`, і якщо крапка зустрічалася в середині шляху (наприклад, у директорії `.web`), вона сприймала все після неї як розширення файлу (наприклад, `.web/apps/3rdparty/industrialbank/cards/public/data/uk/_/t`).

Через це `DBFS` під час завантаження документа без явного розширення вважав, що розширення вже задане, і не застосовував логіку пошуку файлів за резервними розширеннями (`DATA_EXTNAMES`: `.json`, `.yaml` тощо). Відповідно, файли на зразок `t.json` не знаходились. Це критично зламало систему інтернаціоналізації `I18nDb`, оскільки вона передає абсолютний шлях до `t.json` через метод `DBFS.loadDocument()`.

## Що було зроблено
Внесено зміни до функції `extname` у файлі `packages/db/src/DB/path.js`. 
Тепер вона спочатку отримує базове ім'я файлу (basename) за допомогою `uri.split('/').pop()`, і вже з нього витягує розширення. Це повністю усуває помилки парсингу розширень для шляхів, що містять крапки в іменах директорій.

```javascript
export function extname(uri) {
	const base = uri.split('/').pop() || ''
	const arr = base.split('.')
	if (arr.length <= 1 || (arr.length === 2 && arr[0] === '')) return ''
	return `.${arr.pop()}`.toLowerCase()
}
```

## Навіщо це було зроблено
Цей фікс необхідний для того, щоб `DBFS` правильно визначав наявність розширення у файлах, які передаються за допомогою абсолютних шляхів у monorepo (де папка `.web` створює цей крайовий випадок). Це відновлює коректну роботу резервного пошуку `fallbackExt` для `.json`/`.yaml` файлів, і, як наслідок, відновлює роботу `I18nDb` та всього API сервера.
